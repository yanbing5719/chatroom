# epoll
1.只能在linux平台上使用
2.共享内存，省去不必要的拷贝
3.没有最大文件描述符的限制，只受进程可以打开的最大文件数目的限制
# LT和ET的区别
### LT（水平触发）
一般不做特殊说明都是LT模式

> **只要缓冲区还有数据，epoll 就会不断通知你。**
```c
#include <iostream>

#include <sys/socket.h>

#include <arpa/inet.h>

#include <unistd.h>

#include <sys/epoll.h>

  

using namespace std;

int main(int argc, char *argv[])

{

int lfd = socket(AF_INET, SOCK_STREAM, 0);

sockaddr_in addr;

addr.sin_family = AF_INET;

addr.sin_port = htons(2100);

addr.sin_addr.s_addr = INADDR_ANY;

  

int opt = 1;

setsockopt(lfd,

SOL_SOCKET,

SO_REUSEADDR,

&opt,

sizeof(opt));

  

int ret = bind(lfd, (sockaddr *)&addr, sizeof(addr));

ret = listen(lfd, 5);

  

// 创建epoll实例

int epfd = epoll_create(1); // 此处只要是一个大于0的数即可

if (epfd == -1)

{

perror("epoll_creat");

exit(0);

}

struct epoll_event ev;

ev.events = EPOLLIN;

ev.data.fd = lfd;

epoll_ctl(epfd, EPOLL_CTL_ADD, lfd, &ev);

struct epoll_event evs[1024];

int size = sizeof(evs) / sizeof(evs[0]);

while (1)

{

int num = epoll_wait(epfd, evs, size, -1);

cout << num << endl;

for (int i = 0; i < num; i++)

{

int fd = evs[i].data.fd;

if (fd == lfd)

{

int cfd = accept(lfd, nullptr, nullptr);

struct epoll_event ev;

ev.events = EPOLLIN;

ev.data.fd = cfd;

epoll_ctl(epfd, EPOLL_CTL_ADD, cfd, &ev);

}

else

{

char buf[1024];

int len = recv(fd, buf, sizeof(buf), 0);

if(len>0){

cout<<buf<<endl;

}

if (len == 0)

{

epoll_ctl(epfd, EPOLL_CTL_DEL, fd, nullptr);

close(fd); // 最后再进行关闭文件描述符的操作，否则epoll_ctl会返回-1。

continue;

}

}

}

}

close(epfd);

close(lfd);

}
```
### ET（边缘触发）

> **只有数据到来的那一瞬间通知一次。上一次未接受完的数据需要等待下一次数据过来才能发送过来，因此要不断循环接受数据**
```c
#include <iostream>

#include <sys/socket.h>

#include <arpa/inet.h>

#include <unistd.h>

#include <sys/epoll.h>

#include <fcntl.h>

#include <errno.h>

  

using namespace std;

int main(int argc, char *argv[])

{

int lfd = socket(AF_INET, SOCK_STREAM, 0);

sockaddr_in addr;

addr.sin_family = AF_INET;

addr.sin_port = htons(2100);

addr.sin_addr.s_addr = INADDR_ANY;

  

int opt = 1;

setsockopt(lfd,

SOL_SOCKET,

SO_REUSEADDR,

&opt,

sizeof(opt));

  

int ret = bind(lfd, (sockaddr *)&addr, sizeof(addr));

ret = listen(lfd, 5);

  

// 创建epoll实例

int epfd = epoll_create(1); // 此处只要是一个大于0的数即可

if (epfd == -1)

{

perror("epoll_creat");

exit(0);

}

struct epoll_event ev;

ev.events = EPOLLIN;

ev.data.fd = lfd;

epoll_ctl(epfd, EPOLL_CTL_ADD, lfd, &ev);

struct epoll_event evs[1024];

int size = sizeof(evs) / sizeof(evs[0]);

while (1)

{

int num = epoll_wait(epfd, evs, size, -1);

cout << num << endl;

for (int i = 0; i < num; i++)

{

int fd = evs[i].data.fd;

if (fd == lfd)

{

int cfd = accept(lfd, nullptr, nullptr);

// 为了解决recv的阻塞问题，可以完整接收最后一次发送过来的数据，修改文件描述符的属性

int flag = fcntl(cfd, F_GETFL);

flag |= O_NONBLOCK;

fcntl(cfd, F_SETFL, flag);

struct epoll_event ev;

ev.events = EPOLLIN;

ev.data.fd = cfd;

epoll_ctl(epfd, EPOLL_CTL_ADD, cfd, &ev);

}

else

{

char buf[10];

while (1)

{

int len = recv(fd, buf, sizeof(buf), 0);

if (len == -1)

{

if (errno = EAGAIN)

{

cout << "数据传输完毕" << endl;

break;

}

else

{

perror("recv");

exit(1);

}

}

if (len > 0)

{

cout << buf << endl;

}

if (len == 0)

{

epoll_ctl(epfd, EPOLL_CTL_DEL, fd, nullptr);

close(fd); // 最后再进行关闭文件描述符的操作，否则epoll_ctl会返回-1。

continue;

}

}

}

}

}

close(epfd);

close(lfd);

}
```

# Epoll 核心原理总结

## 1. Epoll 解决什么问题

在传统服务器模型中，通常采用 **一个连接对应一个线程** 的方式：

```text
Client1 -> Thread1
Client2 -> Thread2
Client3 -> Thread3
...
```

当客户端数量较少时没有问题，但如果服务器同时维护大量连接：

```text
10000 个客户端
=
10000 个线程
```

会带来以下问题：

* 线程创建和销毁开销大
* 内存占用高
* 线程切换成本高
* CPU资源浪费严重

因此需要一种机制，让少量线程管理大量连接，这就是 Epoll 出现的原因。

---

## 2. Epoll 的核心思想

传统方式：

```text
不停检查所有客户端
```

Epoll方式：

```text
谁有事件
谁通知我
```

即：

```text
事件驱动（Event Driven）
```

服务器不再主动轮询所有连接，而是等待内核通知哪些连接发生了事件。

---

## 3. 一个线程管理多个客户端

Epoll最大的特点之一：

```text
          Epoll Thread
                |
      +---------+---------+
      |         |         |
   Client1   Client2   Client3
```

这里只有：

```text
1 个线程
```

却能够同时管理：

```text
1000
10000
甚至更多连接
```

这也是高并发服务器的基础。

---

## 4. 不发消息会怎样

假设：

```text
Client1
Client2
Client3
```

都已经连接服务器。

主线程执行：

```cpp
epoll_wait(epfd, events, 1024, -1);
```

如果所有客户端都没有发送数据：

```text
Client1 无消息
Client2 无消息
Client3 无消息
```

此时：

```text
Epoll线程被内核挂起
```

CPU几乎不消耗资源：

```text
CPU ≈ 0%
```

因此即使同时在线上万个用户，只要没人发送数据，服务器压力也很小。

---

## 5. 有客户端发消息会怎样

例如：

```text
Client2 发送消息
```

内核检测到：

```text
Client2对应的fd可读
```

然后：

```text
加入就绪队列
```

并唤醒：

```cpp
epoll_wait(...)
```

返回：

```cpp
n = 1;

events[0] = Client2_fd;
```

此时服务器开始处理Client2发送的数据。

---

## 6. Epoll不会返回所有客户端

假设当前状态：

```text
Client1 无消息
Client2 发消息
Client3 无消息
Client4 发消息
Client5 无消息
```

执行：

```cpp
epoll_wait(...)
```

返回结果：

```cpp
n = 2;

events[0] = Client2_fd;
events[1] = Client4_fd;
```

不会返回：

```text
Client1
Client3
Client5
```

因为它们没有事件发生。

因此Epoll真正做到：

```text
只处理活跃连接
```

而不是：

```text
处理全部连接
```

---

## 7. Epoll不负责收发数据

很多初学者容易误解：

```text
Epoll负责网络通信
```

实际上并不是。

Epoll只负责告诉程序：

```text
这个fd可读
这个fd可写
这个fd断开连接
```

真正的数据收发仍然需要程序自己完成：

```cpp
recv(fd, buffer, size, 0);

send(fd, buffer, size, 0);
```

因此：

```text
Epoll负责发现事件

recv/send负责处理数据
```

---

## 8. Epoll内部结构（简化理解）

可以将Epoll理解为：

```text
            Epoll
      +----------------+
      |    红黑树      |
      +----------------+
               |
               v
      +----------------+
      |   就绪队列     |
      +----------------+
```

### 红黑树

作用：

```text
保存所有监听的fd
```

例如：

```cpp
epoll_ctl(epfd,
          EPOLL_CTL_ADD,
          clientfd,
          &ev);
```

所有加入监听的socket都会存放在红黑树中。

---

### 就绪队列

当客户端发送数据时：

```text
Client ---> Server
```

对应fd变为：

```text
可读状态
```

内核会把该fd放入：

```text
就绪队列（Ready Queue）
```

当程序调用：

```cpp
epoll_wait(...)
```

时，直接从就绪队列中取出活跃fd返回。

因此：

```text
连接数再多

只返回真正发生事件的fd
```

这也是Epoll高性能的根本原因。


# Muduo 网络库核心原理

## 什么是 Muduo

一句话概括：

> Muduo 是一个基于 Epoll + Reactor + ThreadPool 的 C++ 高性能网络库。

Muduo帮开发者封装了：

```text
socket
bind
listen
accept
epoll
线程管理
连接管理
缓冲区管理
```

开发者只需要关注：

```text
登录
注册
群聊
私聊
好友
数据库
```

等业务逻辑即可。

---

# Muduo整体架构

```text
                Application
                      |
                      v
                TcpServer
                      |
                      v
                EventLoop
                      |
                      v
                  Poller
                      |
                      v
                   Epoll
```

---

# 核心模块

## 1. EventLoop（事件循环）

Muduo最核心的类。

作用：

```text
不断等待事件
不断处理事件
```

本质上类似：

```cpp
while(true)
{
    epoll_wait(...);

    处理事件;
}
```

因此：

```text
EventLoop
=
epoll事件循环的封装
```

---

## 2. Poller

作用：

```text
封装epoll
```

负责管理：

```cpp
epoll_create()
epoll_ctl()
epoll_wait()
```

在自己实现MyMuduo时：

```text
Epoll类
=
Muduo中的Poller
```

---

## 3. Channel

Muduo最经典的设计之一。

作用：

```text
fd + 事件
```

例如：

```cpp
fd = 10
```

对应：

```cpp
EPOLLIN
EPOLLOUT
EPOLLERR
```

Muduo将其封装为：

```cpp
Channel
```

关系：

```text
一个Channel对应一个fd
```

---

## 4. Acceptor

作用：

```text
专门负责accept
```

内部封装：

```cpp
accept()
```

关系：

```text
listenfd
    ↓
 Acceptor
    ↓
 新连接
```

避免业务层直接操作accept。

---

## 5. TcpConnection

作用：

```text
管理一个客户端连接
```

例如：

```text
Client1
```

对应：

```cpp
TcpConnection
```

内部维护：

```cpp
fd
IP
Port
连接状态
Buffer
```

这是Muduo中非常重要的类。

---

## 6. Buffer

作用：

```text
网络缓冲区
```

避免频繁使用：

```cpp
char buf[1024];
```

Muduo提供：

```cpp
Buffer
```

负责：

```text
接收数据
发送数据
拼包
拆包
```

---

## 7. ThreadPool

作用：

```text
线程池
```

负责：

```text
处理业务逻辑
```

例如：

```text
登录
注册
私聊
群聊
数据库操作
```

---

# 总结

Muduo本质上解决的是：

```text
网络通信
连接管理
事件分发
线程管理
```

而不负责：

```text
登录
注册
好友
群聊
私聊
数据库
```

这些业务逻辑。

因此：

```text
Muduo负责网络层

聊天室负责业务层
```

自己实现MyMuduo时，可以先完成聊天室项目，再逐步将：

```text
Epoll
ThreadPool
Connection
EventLoop
```

抽象出来形成独立网络库。
